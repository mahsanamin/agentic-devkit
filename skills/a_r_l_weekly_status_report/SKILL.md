# a_r_l_weekly_status_report

A two-step routine: **first reconcile** a task-flow workspace against reality with the progress-fixer, **then write** the weekly report off the corrected state. Run it from the workspace directory.

**Step 1 is mandatory and always runs first — never skip it and never let the report go first.** People routinely forget to move their tasks as PRs merge (a merged ticket left in `OnGoingTasks/`, a missing completion date), so the workspace drifts from reality, and a report written off the drifted state mislabels shipped work as in progress and vice versa. The fixer's read-only reconciliation (scan → check → report) is what corrects the picture the report is built on, so it can never be skipped, even when the workspace looks tidy. A report is only as trustworthy as the reconciliation that preceded it.

Run autonomously; the only place you pause is the explicit approval gate below, which covers **only** the fixer's **destructive** auto-fixes (archiving folders, filling dates), not the read-only reconciliation and not the report. Generating, committing, and pushing the report file itself is **not** gated: it is the deliverable, it is additive, and it always happens. **The run is not complete until the report is committed and pushed to the remote.**

## Inputs (fill in the blanks)

| Input | Meaning | Example |
|-------|---------|---------|
| `workspace_dir` | The task-flow workspace to run from. | `Coding_Tasks/` |
| `audience` | Who the report is for (sets tone + altitude). | CEO and CTO |
| `report_dest` | Where the report file is written. | the docs repo's `ProjectStatus/SharedWeeklyReports/` |

Each is `{id, name, kind}`. | for a project: the code/dev channel + the product channel, configured in `config` |

A typical workspace: per-platform task folders under `Backend/` and `Frontend/` as `OnGoingTasks/` and `DoneTasks/`; status rows in `TasksSummary/<Platform>.md`; config in `config` (namespace, GitHub repos per platform, release branches like `develop` + `release/*`).

## The flow (in order)

1. **`fixer_skill` (`/aa-task-flow-progress-fixer`) — always run this FIRST, every run, no exceptions.** Invoke it explicitly by name; it sets `disable-model-invocation`, so it will not fire on its own and the routine must call it. It reconciles every `OnGoingTasks` folder against actual git/GitHub state: verify PR merge status per ticket, find Phase-4 tasks merged to a release branch that should be archived, surface stale summaries and branch/worktree drift, and flag completion-date gaps in `TasksSummary`. This scan → check → report pass is **read-only** and is the whole reason it goes first: it produces the corrected, reality-checked view of every task that step 2 reports against. **Only the auto-fix step is gated** — show the reconciliation report and get explicit approval before applying any destructive edit (archiving folders, filling dates). On approval, commit + push those fixer changes to the docs repo. If you held for approval (e.g. an unattended run), skip *applying* the fixes but carry the reconciled findings forward into step 2 anyway: the report is still generated and pushed, and it still classifies against the real merge state the fixer just established, not against where folders happen to sit.
2. **`report_skill`** — once the fixer has reconciled the workspace (step 1 always runs first), generate the weekly report for the week just ended. Audience is `audience`. Read the project state file, the two prior reports, and this-week task evidence; classify each item (Shipped / Development Done / In progress / Concerns / Next) against real merge state; keep engineering plumbing out of the copy. Save to `report_dest`. (Commit + push happens after step 3.)
3. **Slack hint pass (read-only) — once the report is drafted, cross-check it against `slack_channels` for the report's date range.** Resolve the channels from `config` (`weekly_report.slack_channels`); if none are configured, skip this step and say so. For each channel, read history over the report window plus the few days since (so late updates are caught), e.g. `conversations_history` with a date-range limit. Then **lift only the signals that the code cannot show**: a feature rolled back after merge, a scope or priority change, a launch-timing or green-light call, a leadership concern, real production-usage numbers, a decision that supersedes what a PR implies. **The code is the exact picture; Slack only adds hints** — never let a Slack message override the merge-state classification from step 2, never invent status from chatter, and fold a hint in only when it changes or sharpens what the report says. Keep the executive voice; attribute lightly (e.g. "per team Slack") where a claim is conversational rather than code-verified. **File each hint under the section it actually belongs to, and respect what each section means.** In particular, **SLOs are reliability metrics only** (availability, latency, error rate — actual vs target with a RAG status); a product, usage, traction, or adoption number is **not** an SLO and must never go there — put it with the relevant Shipped item's impact/evidence, or in the status line. Match every other hint to its real home too: a post-merge rollback → Concerns (or the affected In-progress/Shipped item), a priority or scope change → Next, a launch-timing call → status/Next. **This pass is read-only on Slack: never post a message** (see the global "never send Slack directly" rule).

Then **commit and push** the report to the docs repo. This commit is mandatory and is **not** behind the approval gate. Stage only the report file (e.g. `git add "<report_dest>/<file>.md"`, plus any fixer changes already approved in step 1) and **never `git add -A`**, so unrelated untracked files (stray task folders, temp notes) are not swept in. A run that generated the report but left it uncommitted is **not** complete: a saved-but-unpushed file is the failure mode this step exists to prevent.

This flow fixes the progress of items first, produces a clean report off the corrected state, then layers in the conversational hints that only Slack carries.

## Gotchas worth knowing before you run it

- **Never skip the fixer, and never let the report run first.** The most common failure this routine exists to catch is a colleague who merged a PR but never moved the task to `DoneTasks/` (or never filled the completion date). The workspace can look perfectly tidy and still be wrong, so "it looks fine" is never a reason to skip step 1. The report is only as accurate as the reconciliation that preceded it.
- **Get explicit approval before archiving.** The fixer can move ~20 folders and edit `TasksSummary` in one pass. Those destructive moves are shared-state changes, so confirm first. This is the one approval gate in an otherwise-autonomous routine, and it covers the fixer's archiving and date edits **only**. It does **not** gate the fixer's read-only reconciliation (that always runs first and always feeds the report), and it does **not** gate the report's own commit + push (see the Report step). So even an unattended run that holds the archiving still reconciles state and still pushes a report classified against real merge state.
- **Mind which branches count as released.** Work merged to a branch that is NOT in `release_branches` (e.g. `staging`) classifies as *In progress*, not Development Done. Decide per run whether that's right, or add the branch to `release_branches` in `config` if it is a promotion branch.
- **`project_repo_root` may be unset**, which makes the fixer skip branch-existence and worktree-mapping checks. Set it in `config` to enable them.
- **The shell is zsh.** Unquoted `$var` does not word-split. Loop over arrays (`for t in "${arr[@]}"`), not space-joined strings.
- **No "Shipped" while a launch is gated.** If the public still sees a waitlist, code merged to the main release branch is deployed but gated, so it classifies as *Development Done — awaiting public cutover*, not *Shipped*.
- **Slack is a hint layer, not a source of truth, and the channels are config not code.** Git/PR state is the exact picture; the Slack pass only surfaces the small things code can't show (a post-merge rollback, a priority flip, a green-light/launch-timing call, a leadership concern, live usage numbers). Read the channel list from `config` (`weekly_report.slack_channels`) so it stays project-agnostic — never bake channel IDs into the skill, and skip the pass cleanly when none are configured. The pass is **read-only**: read history, never post.
- **A hint goes in the section it belongs to — and SLOs are reliability-only.** The SLO section is availability/latency/error-rate vs target, a fixed line copied from prior reports; a product/usage/traction/adoption number (e.g. "N users started a signup") is **not** an SLO. Put usage signals with the relevant Shipped item's impact/evidence or in the status line, never under SLOs. (This is the exact mistake to avoid: a Slack usage number was once dropped into SLOs — it belongs with the shipped item it measures.)

## Report

The deliverable is the weekly report file at `report_dest`, reconciled against git first, layered with read-only Slack hints, and **committed and pushed** to the docs repo remote. The run only counts as complete once that push lands. End the run by stating: what the fixer changed (or that you held for approval), which Slack hints (if any) you folded in and from which channels, the path to the generated report, and the commit SHA + branch it was pushed on. If the push failed or was skipped, say so loudly and treat the run as incomplete.
